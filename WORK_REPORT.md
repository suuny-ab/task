# DataLens AI — 工作报告

## 项目概述

DataLens AI 是一个 AI 驱动的数据分析平台，面向"底表 → 规则 → 结论表 → 逐层下钻解读"的通用业务场景（销售奖金核算、直播达人分佣、物流时效分析等）。

**项目起点：** 一套 UI 高保真原型，交互流程完整，但所有数据和 AI 能力均为硬编码 Mock。

**部署地址：** [https://task-vert-chi-61.vercel.app](https://task-vert-chi-61.vercel.app)

---

## 功能选择与决策

### 选择实现：AI 下钻解读接入 LLM（P0）

**选择理由：**

1. **与大模型应用岗位最匹配** — LLM 集成、Prompt 工程、流式交互设计正是该岗位的核心能力
2. **展示效果最直观** — 打开页面 → 点击结论表任意一行 → 右侧面板 AI 实时流式生成数据解读，无需额外说明
3. **改动可控** — 保留原型的 Mock 数据结构（values、children、rules），仅将硬编码的 `aiSummary` 替换为 LLM 实时生成，改动集中、风险低

**未选择的功能：**

| 功能 | 不选理由 |
|------|---------|
| Excel 解析 | 偏基础设施，与 AI 岗位关联弱 |
| 动态计算引擎 | 工作量大，需实现完整计算逻辑，1 天不可控 |
| 数据库持久化 | 非 AI 能力 |
| 用户认证 | 与岗位无关 |

## 实现方案

### 整体架构

```
用户点击结论表某一行
      ↓
page.tsx: fetchAISummary(node)
      ↓
POST /api/drilldown  ← Next.js API Route (Serverless Function)
      ↓
prompt-builder.ts: 判断节点类型 → 组装 Prompt
      ↓
DeepSeek API (流式调用, OpenAI 兼容协议)
      ↓
ReadableStream → 前端逐字渲染
```

### 技术选型

| 决策项 | 选择 | 理由 |
|--------|------|------|
| LLM | DeepSeek | 国产模型，API 兼容 OpenAI 格式，中文能力强，成本低 |
| SDK | `openai` JS SDK | DeepSeek 完全兼容 OpenAI 协议，无需额外适配 |
| 后端 | Next.js API Route | 与前端同项目，部署到 Vercel 自动变为 Serverless Function |
| 响应方式 | 流式 (Streaming) | 避免用户等待空白，实现逐字打字效果 |

### Prompt 工程设计

**核心思路：** 通过分析三个场景（销售奖金、直播分佣、物流时效）的下钻树结构，提炼出场景无关的共性模式：

- 非叶子节点（有子节点）：侧重**指标关系 + 子节点构成分析**
- 叶子节点（无子节点）：侧重**计算公式 + 规则引用**

**关键设计决策：**

| 决策 | 方案 | 理由 |
|------|------|------|
| Prompt 分类 | 按节点类型分两套模板 | 比单一 Prompt 输出更稳定可控 |
| 业务通用性 | Prompt 不含任何业务场景词汇 | 维度名、指标名全部来自数据本身，AI 自行推断场景 |
| 语体风格 | 数据分析报告的书面语体 | 面向 B 端业务人员，简练、精准、克制 |
| 节点类型判定 | `node.children?.length > 0` → branch | 代码侧判定，而非让 AI 自己判断 |

**非叶子节点 System Prompt：**

```
你是企业数据分析平台的 AI 解读引擎。用户提供一个数据节点的结构化信息，你需要生成简明的分析摘要。

风格要求：
- 采用数据分析报告的书面语体，简练、精准、克制
- 读者是 B 端业务人员，熟悉本业务领域的基本概念和指标含义，无需解释常识
- 用数字说话：所有提供的指标均须在摘要中体现其数值，不得遗漏
- 先总后分：先概述该节点的全部指标（含核心结论指标和过程指标），再说明子节点构成
- 如有规则影响结果，简要点明规则名称和生效条件，不要展开解释规则的含义
- 控制在 3-5 句话，每句话都应传递新信息，杜绝冗余和重复
- 直接输出分析文本，禁止任何开场白、标题、序号
```

**叶子节点 System Prompt：**

```
你是企业数据分析平台的 AI 解读引擎。用户提供一个最细粒度的数据节点，含具体计算公式和适用规则，你需要生成简明的计算说明。

风格要求：
- 采用数据分析报告的书面语体，简练、精准、克制
- 读者是 B 端业务人员，熟悉本业务领域的基本概念，无需解释常识
- 陈述原始数据 → 点明适用规则及生效条件 → 展示公式代入结果，一气呵成
- 控制在 2-3 句话，杜绝冗余和重复
- 直接输出分析文本，禁止任何开场白、标题、序号
```

**User Prompt 数据组装：** 将 `DrilldownNode` 的结构化字段（label、levelLabel、values、children 摘要、rules、lineage）序列化为统一文本格式，数值预先格式化为可读字符串（如 `5280000` → `528.00 万`），避免 AI 自行格式化出错。

## AI 输出效果对比

以"销售奖金 → 张三"三层下钻为例，对比 Mock 硬编码与 AI 实时生成的解读：

### 第一层：张三（销售维度，Branch 节点）

**Mock 硬编码：**
> 张三本期总营收 528.00 万，经产品系数折算后有效业绩为 452.00 万，按 2.5% 的奖金比例，最终应发奖金 11.30 万。该销售共服务 5 个客户，以下为按奖金贡献排序的客户明细：

**AI 生成：**
> 张三总营收528.00万，经产品系数规则折算后有效业绩为452.00万，达到400万以上分档，适用2.5%的奖金比例，应发奖金为11.30万。其奖金主要由前三大客户贡献，其中上海云途科技贡献3.30万，杭州数联信息贡献2.13万，苏州智算科技贡献1.55万。

### 第二层：上海云途科技（客户维度，Branch 节点）

**Mock 硬编码：**
> 上海云途科技有限公司为张三贡献了 3.30 万奖金，占其总奖金的 29.2%。该客户消费总额 158.00 万，经不同产品系数折算后有效业绩为 132.00 万。

**AI 生成：**
> 该客户总消费158.00万元，产生有效业绩132.00万元，最终贡献奖金3.30万元，占奖金池总额的29.2%。奖金贡献主要来源于三项产品：云服务器ECS贡献1.45万元，对象存储OSS贡献6,400元，云数据库RDS贡献5,950元。

### 第三层：云服务器 ECS（产品维度，Leaf 节点）

**Mock 硬编码：**
> 云服务器 ECS 在客户上海云途科技有限公司的消费中贡献了 58.00 万营收。该产品的业绩计算系数为 1.0（核心计算产品），折算后有效业绩为 58.00 万。

**AI 生成：**
> 云服务器ECS原始消费58.00万元，适用核心计算产品系数1.0的规则，折算后有效业绩为58.00万元。代入公式：奖金贡献 = 58.00万 × 1.0 × 2.5% = 1.45万元。

### 对比小结

| 维度 | Mock | AI |
|------|------|-----|
| 数据准确性 | 准确 | 准确（数字全部正确） |
| 因果归因 | 只列数字 | 点明规则档位和生效条件 |
| 构成分析 | 无 | 主动引用子节点数据，给出占比 |
| 规则引用 | 无 | 引用规则名称和适用条件 |
| 语体 | 报表式平铺 | 分析报告式，先总后分 |

## 通用性验证

同一套 Prompt 模板在三个完全不同的业务场景下均能正确工作，无需修改任何业务词汇：

| 场景 | 下钻路径 | AI 表现 |
|------|---------|---------|
| 销售奖金 | 张三 → 上海云途科技 → 云服务器 ECS | 正确陈述营收、有效业绩、奖金比例档位、客户构成、产品系数 |
| 直播分佣 | 李佳琦 → 情人节专场 → 兰蔻小黑瓶 | 正确陈述 GMV、退货率、有效 GMV、分佣比例、场次贡献 |
| 物流时效 | 华东仓 → 上海徐汇店 → 次日达批次 | 正确陈述配送单量、达标率、平均耗时、门店排名 |
